# B站评论雷达

帮助用户把“想从评论区里找有用信息，但刷不动、找不准、容易漏掉楼中楼”的困扰，转成**可读的结论 + 可核对的评论原文**。目标不是替用户“编造观点”，也不是简单按点赞排序，而是在合理抓取范围内，尽量找出与用户问题语义最匹配的评论证据。

## 核心功能

- **单视频评论检索**：针对一条视频，按用户问题从评论池里筛出最相关的若干条。
- **分页抓取与去重**：主评论分页拉取，减少重复条目干扰。
- **深度楼中楼**：在需要时继续翻子回复页，降低“关键意见藏在楼里”的遗漏概率。
- **混合排序**：结合语义向量相似度与关键词匹配；可选开启 LLM 二次重排（需环境已配置）。
- **结论先行输出**：先回答用户问题，再列出证据评论，方便决策与复核。

## 典型触发场景

当用户出现以下需求时，应优先使用本 Skill：

- “这个视频评论区里有人提到 X 吗？”
- “大家对这个产品/课程/观点的真实反馈是什么？”
- “值不值得买/学/跟做？我想看评论依据”
- “关于某个细节（续航、噪音、难度、翻车）评论区怎么说？”
- “热门评论不回答我的问题，帮我找更相关的几条”
- “结果太少/不够准，扩大范围或提高精度再试一次”

## 适用场景

- 数码选购、测评类视频下的真实体验与避坑线索
- 教程、课程类视频下的难度、劝退、跟学建议
- 社会热点或争议话题类视频下的观点分布（以评论证据呈现，不下武断定论）
- 需要快速核对“评论区有没有某种说法”的轻量调研

## 输入方式

用户可以用自然语言触发，本 Skill 需要兼容以下输入：

### 1. 链接 + 明确问题（最推荐）

- “帮我在这个视频评论里找和「续航虚标」最相关的 5 条，先结论后证据：`<视频链接>`”
- “`BVxxxxxx`，评论区对「新手能不能学」怎么说？”

### 2. 先给链接，再补充问题

- 第一轮：`https://www.bilibili.com/video/BV...`
- 第二轮：“同上，我想看和「噪音」相关的评论，Top 8。”

### 3. 只给 BV 或链接，问题较笼统

- “分析一下这个视频评论区都在吵什么。”  
  此时应先请用户收窄问题，例如：更关心选购、学习难度，还是某个具体槽点。

### 4. 指定输出或策略偏好

用户也可能补充：

- **数量**：Top 3 / Top 5 / Top 10
- **侧重点**：优先楼中楼、优先高赞为辅、只要“翻车/劝退”相关
- **速度与覆盖**：更快出结果 vs 更全面、更怕漏
- **精度**：是否需要开启更高精度重排（依赖 LLM 配置）

## 工作流程

当本 Skill 被触发时，按以下顺序处理：

### Step 1：确认视频与问题

- 从用户消息中提取 `bvid_or_url`（链接或 BV）。
- 提取或澄清 `query`。若过于宽泛，先用 1～2 句反问收窄范围。

### Step 2：选择默认抓取与排序策略

- 常规：分页抓取主评论 + 按需深挖楼中楼。
- 默认倾向：**先覆盖再精排**；用户强调速度时再收紧抓取。

### Step 3：调用检索能力

- 优先使用 `search_similar_comments_tool` 完成“抓取 + 相似度排序 + TopN”。
- 若用户需要审计抓取范围、导出原始评论列表，再使用 `fetch_video_comments_tool`。

### Step 4：合成回答

- **结论**必须可被评论证据支撑；证据不足时降低确信度，并说明限制。
- 列出 TopN 时带上相关度、简述原因；点赞仅作辅助参考。

### Step 5：结果不理想时的升级路径（按顺序尝试）

1. 扩大主评论页数（`max_pages`）。
2. 确保/加强深度楼中楼（`deep_fetch_replies` 与单楼分页上限）。
3. 在环境允许时开启 LLM 重排（`enable_llm_rerank`）。

### Step 6：收尾提示

- 简要说明本次样本规模（如工具返回的 `candidate_count`）与结论可信度。
- 给用户一句**可复制的下一步话术**（例如如何要求“再扩大抓取”）。

**默认工具参数参考（可在对话中按需调整）：**

- `top_n`: `5`
- `max_pages`: `20`
- `sort`: `hot`
- `include_replies`: `true`
- `deep_fetch_replies`: `true`
- `max_reply_pages_per_root`: `5`
- `max_total_replies`: `1000`
- `rerank_mode`: `hybrid`
- `embedding_model`: `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`
- `enable_llm_rerank`: `false`（需要更精细且已配置 API 时再为 `true`）

**策略速查：**

- **速度优先**：`deep_fetch_replies=false`，`max_pages` 降至 `8~10`
- **覆盖优先**：提高 `max_pages`、`max_reply_pages_per_root`、`max_total_replies`
- **精度优先**：`enable_llm_rerank=true`，可适当提高 `llm_rerank_top_k`

## 输出格式

默认使用以下结构输出：

```markdown
#### 结论
[一句话回答用户问题；若证据弱，需标明“弱相关/样本不足”]

#### 证据评论（TopN）
1) [相关度 {score}] {comment_text}
   - 原因：{reason}
   - 点赞：{like}

2) [相关度 {score}] {comment_text}
   - 原因：{reason}
   - 点赞：{like}

#### 可信度与下一步（可选）
- 当前评论样本规模：{candidate_count}
- 结论可信度：高 / 中 / 低
- 建议下一步：{例如扩大抓取、开启高精度重排、或把问题改得更具体}
```

若用户只要极简版，可只输出「结论 + 3 条证据」，但仍须保留可核对原文。

## 输出原则

始终遵循以下原则：

1. **证据优先**  
   结论必须能落到具体评论原文上；没有足够证据就不做强结论。

2. **不编造**  
   不伪造评论内容、不杜撰用户名或点赞数据。

3. **点赞为辅**  
   高赞只作辅助信号，不能替代与用户问题的相关性。

4. **楼中楼与覆盖面**  
   对“怕漏细节”的需求，应主动倾向更深子回复抓取（在用户可接受的耗时内）。

5. **语义与关键词兼顾**  
   既照顾同义表达（语义），也照顾显性关键词命中。

6. **风险表述克制**  
   涉及人身攻击、阴谋论、未经核实传言等，只作归纳呈现并提示谨慎采纳。

7. **可执行下一步**  
   抓取不足或排序不稳时，给出用户能直接复制使用的跟进指令。

## 示例

### 示例 1：买不买心里没底

**用户咋说：**  
刷到一个测评，懒得翻评论了。链接在这儿，你帮我看看有没有人吐槽续航造假或者缩水，有的话挑几条最像那么回事的给我，顺带说一句评论区总体是啥风向。  
`https://www.bilibili.com/video/BVxxxxxx`

**回复时注意：**  
先用大白话总结评论区大概站哪边；再把几条原评论贴出来，别编。

### 示例 2：教程怕跟不下来

**用户咋说：**  
`BVxxxxxx` 这个课我想跟，但我是纯小白。你翻翻评论，有没有人说太贵了、太劝退、跟不上之类的？多给几条，尤其是楼里吵得比较细的。

**回复时注意：**  
别只盯着热评，楼里提到的难点可以一起拎出来。

### 示例 3：觉得给少了

**用户咋说：**  
就这几条？不行，再多捞点，楼中楼也刷刷，最后还是给我个五六条最相关的就行。

**回复时注意：**  
把抓取范围加大再跑一趟；如果样本多了，跟用户说一句大概扫了多少条评论（有工具里的数就用）。

### 示例 4：觉得排得不对

**用户咋说：**  
你这排的啥啊，我问的是噪音大不大，往前几条都不太沾边。能不能再精细筛一遍？

**回复时注意：**  
能开高精度重排就开，开不了就老实说，按现在的混合排序再试，顺便让用户把问题说窄一点（比如只关心「夜间噪声」）。

## 注意事项

1. 遵守平台规则与合理请求频率，避免高频滥用接口。
2. 本 Skill 输出的是**基于已抓取评论的证据整理**，不是全站统计或官方结论。
3. 对争议话题避免替用户做法律/道德裁决，以“评论里怎么说”为边界。
4. 若抓取失败或评论关闭等，应如实说明原因，不要编造占位评论。
5. 涉及医疗健康、投资理财等高风险领域，应提示用户以专业渠道为准。
